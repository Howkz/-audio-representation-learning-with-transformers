\section{Introduction}
Ce rapport présente une étude expérimentale progressive sur un encodeur Transformer audio pré-entraîné par MAE puis adapté en ASR via CTC.
L'objectif scientifique est triple: (i) vérifier l'apport du pré-entraînement MAE, (ii) identifier les compromis entre WER, temps d'inférence et mémoire GPU, (iii) produire un benchmark multi-variantes avec consolidation statistique finale.

Contraintes de conception: environnement vdigpu, execution reproductible, suivi frugal des ressources, contrainte dure de temps (24h), et budget stockage traite comme contrainte locale variable.

\paragraph{Choix globaux assumés}
\begin{itemize}
\item Screening uniquement sur LibriSpeech pour tenir le budget disque/temps.
\item VoxPopuli est exclu par défaut de cette suite (extension possible si marge).
\item Le cache data est conservé entre expériences.
\item Le budget local de stockage (par ex. 18 Go) pilote l'execution locale, mais n'est pas encode comme limite scientifique fixe.
\end{itemize}


\section{Questions de recherche et hypothèses globales}
\subsection{Questions de recherche}
\begin{enumerate}
\item \textbf{RQ1}: Le pré-entraînement MAE améliore-t-il la performance ASR (WER) par rapport à un entraînement CTC sans pré-entraînement ?
\item \textbf{RQ2}: Quel compromis est obtenu entre qualité (WER), coût temporel (runtime/débit) et coût matériel (mémoire GPU) selon la capacité du modèle ?
\item \textbf{RQ3}: Quels choix d'architecture (positionnel, stratégie de patching) et d'hyperparamètres MAE (mask ratio) sont les plus robustes ?
\item \textbf{RQ4}: Les conclusions restent-elles stables sous variabilité aléatoire (5 seeds sur les meilleures variantes) ?
\end{enumerate}

\subsection{Hypothèses globales justifiées}
\begin{itemize}
\item \textbf{H1 (apport MAE)}: une représentation auto-supervisée pré-entraînée améliore la généralisation ASR, car l'encodeur apprend des régularités acoustiques indépendantes de l'alignement texte.
\item \textbf{H2 (capacité)}: augmenter la capacité du Transformer peut réduire le WER, mais au prix d'une hausse de runtime et de mémoire.
\item \textbf{H3 (inductive bias)}: des choix de patching et d'encodage positionnel modifient le biais inductif; certains réglages peuvent mieux capturer la structure temps-fréquence.
\item \textbf{H4 (mask ratio)}: le ratio de masquage MAE gouverne la difficulté de prétexte; trop faible sous-contraint l'apprentissage, trop fort peut dégrader l'optimisation.
\end{itemize}


\section{Méthodologie générale}
\subsection{Définitions opératoires}
\begin{itemize}
\item \textbf{ASR (Automatic Speech Recognition)}: conversion d'un signal de parole en transcription texte.
\item \textbf{CTC (Connectionist Temporal Classification)}: fonction de perte permettant d'apprendre sans alignement frame-à-frame explicite.
\item \textbf{WER (Word Error Rate)}: métrique de transcription basée sur substitutions, insertions et suppressions.
\item \textbf{MAE (Masked AutoEncoder)}: pré-entraînement auto-supervisé par reconstruction de portions masquées du signal.
\item \textbf{TTS (Text-To-Speech)}: tâche inverse de l'ASR (texte vers audio), non incluse dans le cœur de cette suite.
\end{itemize}

\subsection{Pipeline}
Le protocole suit les étapes \texttt{make data}, \texttt{make train}, \texttt{make test}.
Chaque expérience est exécutée dans un namespace dédié, avec nettoyage des checkpoints intermédiaires après archivage des résultats, tout en conservant le cache de données.


\subsection{Compromis d'implementation audio (absence de torchaudio)}
L'environnement vdigpu cible n'expose pas toujours \texttt{torchaudio} (et le decodeur \texttt{torchcodec} associe a \texttt{datasets} en streaming).
Le pipeline a donc ete adapte pour rester executable:
\begin{itemize}
\item chargement dataset en mode streaming avec desactivation du decodage audio natif \texttt{datasets};
\item decodage audio applicatif via \texttt{soundfile} lorsque disponible, avec fallback \texttt{ffmpeg};
\item execution DataLoader en \texttt{num\_workers=0} dans les profils frugaux pour eviter les crashes workers.
\end{itemize}
Ces choix maintiennent la faisabilite experimentale sous contrainte disque/temps, au prix de compromis explicites:
\begin{itemize}
\item surcout CPU potentiel au decodage, donc debit inferieur a une stack 100\% \texttt{torchaudio};
\item variabilite possible de latence d'IO selon l'outil de decodage effectivement present;
\item differences mineures possibles de re-echantillonnage entre backends audio.
\end{itemize}
\subsection{Politique dataset et périmètre}
\begin{itemize}
\item \textbf{Screening}: librispeech\_only.
\item \textbf{Consolidation finale}: librispeech\_only.
\item VoxPopuli reste une extension possible, hors chemin critique de la présente étude (contraintes disque/temps).
\end{itemize}

\subsection{Politique dataset calibree par phase}
Le respect du budget 24h est porte par des choix fixes de quotas et d'hyperparametres:
\begin{itemize}
\item Screening (E01--E08): \texttt{pretrain=900}, \texttt{asr\_train=1100}, \texttt{asr\_valid=300}, 1 seed.
\item Selection (SEL01--SEL05): memes quotas, 2 seeds.
\item Finale (E09--E11): sous-echantillon elargi fixe (\texttt{pretrain=2000}, \texttt{asr\_train=3000}), 5 seeds.
\end{itemize}
Ainsi, le protocole ne depend pas de variables globales de budget; il depend de configs experimentales explicites.

\subsection{Contrôle expérimental}
\begin{itemize}
\item Même pipeline de prep rocessing audio pour toutes les variantes.
\item Même protocole d'évaluation et mêmes splits pour la comparabilité.
\item Une seule variable manipulée à la fois durant le screening autant que possible.
\item Checkpointing complet (modèle, optimiseur, scheduler, scaler AMP, état RNG) pour reprise fiable.
\end{itemize}

\subsection{Métriques}
Les métriques principales sont:
\begin{itemize}
\item WER (Word Error Rate) pour la performance ASR.
\item Temps d'inférence et débit (samples/s) pour la frugalité temporelle.
\item Mémoire GPU pic pour la frugalité matérielle.
\item Taille modèle (\#paramètres) pour l'analyse capacité/coût.
\end{itemize}

\subsection{Règles statistiques}
Le screening initial est réalisé en 1 seed par variante (E01--E08).
La phase de sélection intermédiaire est réalisée sur Top-5 en 2 seeds (SEL01--SEL05).
La consolidation finale est réalisée sur 5 seeds pour les 3 meilleures variantes (E09--E11), avec reporting moyenne~$\pm$~écart-type.

\subsection{Règle de décision Top-3}
Le classement screening est effectué par tri lexicographique sur:
\begin{enumerate}
\item WER (ascendant),
\item runtime d'inférence (ascendant),
\item mémoire GPU pic (ascendant).
\end{enumerate}

\subsection{Plan de pilotage 24h et regles anti-derive}
\begin{itemize}
\item Decoupage cible: E00 (smoke), E01--E08 (screening), SEL01--SEL05 (selection), E09--E11 (final).
\item Si ETA $>$ 24h apres E04: reduction de \texttt{finetune.max\_steps} de 15--20\% pour E05--E11.
\item Si ETA $>$ 24h apres selection: reduction uniquement de \texttt{pretrain.max\_steps} en finale.
\item Interdit en finale: changer les seeds finales ou les splits de test.
\end{itemize}

\subsection{Interface d'execution unique}
La suite est pilotee par un script Linux unique expose avec les modes:
\texttt{smoke | suite | resume | dry-run | clean}.

Parametrage public:
\texttt{EXPERIMENT\_CACHE\_ROOT}.
Les contraintes de temps/stockage sont gerees par le design des configurations (steps, quotas, seeds), pas par variables globales de budget.

Artefacts attendus:
\texttt{leaderboard\_screening.csv},
\texttt{leaderboard\_selection.csv},
\texttt{suite\_summary.csv},
configs runtime et resultats \texttt{partial/final}.

\subsection{Matrice parametrique de reference (E01) et justification}
Tous les choix non modifies explicitement dans une experience heritent de ce profil de reference.

\paragraph{Parametres audio}
\begin{itemize}
\item \texttt{sample\_rate=16000}: standard ASR, compromis qualite/cout, compatible datasets cibles.
\item \texttt{max\_duration\_sec=6.0}: borne la longueur sequence pour reduire OOM et variabilite de runtime.
\item \texttt{n\_mels=80}: resolution frequencielle suffisante pour parole sans surcout excessif.
\item \texttt{win\_length=400}, \texttt{hop\_length=160}: fenetrage 25ms/10ms, pratique standard robuste en ASR.
\end{itemize}

\paragraph{Parametres modele (baseline small MAE)}
\begin{itemize}
\item \texttt{dim=192}, \texttt{depth=4}, \texttt{num\_heads=6}, \texttt{mlp\_ratio=3.0}: capacite intermediaire pour tenir le budget 24h tout en restant competitive.
\item \texttt{dropout=0.1}: regularisation moderee pour limiter sur-apprentissage sur sous-echantillons.
\item \texttt{patch\_strategy=time}, \texttt{patch\_time=4}, \texttt{patch\_freq=80}: patching simple et frugal, reference pour comparer E06.
\item \texttt{max\_len=1536}: limite memoire explicite, stabilite batch.
\item \texttt{pos\_embed=sinusoidal}: base deterministe et stable, comparee a learned en E05.
\item \texttt{mae\_mask\_ratio=0.6}: niveau de difficulte intermediaire, compare a 0.4 (E07) et 0.75 (E08).
\item \texttt{mae\_decoder\_dim=96}, \texttt{mae\_decoder\_depth=1}, \texttt{mae\_decoder\_heads=4}: decodeur volontairement leger pour frugalite pretrain.
\end{itemize}

\paragraph{Parametres optimisation et entrainement}
\begin{itemize}
\item \texttt{amp=true}: reduction memoire + acceleration, contrainte vdigpu.
\item \texttt{grad\_clip\_norm=1.0}: stabilisation des gradients sur runs longs.
\item \texttt{pretrain}: \texttt{epochs=1}, \texttt{batch\_size=4}, \texttt{grad\_accum\_steps=2}, \texttt{lr=1e-4}, \texttt{weight\_decay=0.01}, \texttt{max\_steps=140}.
\item \texttt{finetune}: \texttt{epochs=1}, \texttt{batch\_size=2}, \texttt{grad\_accum\_steps=4}, \texttt{lr=1.5e-4}, \texttt{weight\_decay=0.01}, \texttt{max\_steps=200}.
\item Justification globale: optimiser le ratio signal/cout pour parcourir E00--E11 dans la contrainte 24h.
\end{itemize}

\paragraph{Parametres data et seeds par phase}
\begin{itemize}
\item Screening E01--E08: \texttt{pretrain=900}, \texttt{asr\_train=1100}, \texttt{asr\_valid=300}, \texttt{asr\_test=300}, seed \texttt{[42]}.
\item Selection SEL01--SEL05: memes quotas, seeds \texttt{[42, 123]}.
\item Finale E09--E11: \texttt{pretrain=2000}, \texttt{asr\_train=3000}, validation/test non tronques si budget disque effectif le permet, seeds \texttt{[42, 123, 456, 789, 1024]}.
\end{itemize}

\paragraph{Regles d'adaptation temporelle (sans changer le protocole statistique)}
\begin{itemize}
\item Si ETA $>$ 24h apres E04: reduire \texttt{finetune.max\_steps} de 15--20\% sur E05--E11.
\item Si ETA $>$ 24h apres selection: reduire seulement \texttt{pretrain.max\_steps} en finale.
\item Interdit: changer seeds finales ou split de test final.
\end{itemize}

\section{Expérience E06 --- Small Time-Frequency Patching}
